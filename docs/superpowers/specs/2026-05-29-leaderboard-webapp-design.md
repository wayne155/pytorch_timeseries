# Leaderboard Webapp — Design Spec

**Date:** 2026-05-29

## Goal

A deployable leaderboard webpage for comparing time series models across tasks, datasets, and hparam configurations. Supports WandB-style filtering, per-seed result storage, and a client-side aggregate/individual-seed toggle. Deployable as a static site (GitHub Pages) or a live local server.

---

## Data Pipeline: `scripts/build_leaderboard.py`

### Inputs

1. **`results/*.json`** — one file per (model, task, dataset, seed) run. Fields used:
   - `model`, `task`, `dataset`, `seed`
   - `hparams` (dict — may include any keys)
   - `metrics` (dict of `{metric_name: float}`)
   - `num_params`, `train_time_sec`, `git_commit`, `timestamp`

2. **`leaderboard/entries/*.yaml`** — curated entries. Each entry:
   ```yaml
   - model: DLinear
     task: Forecast
     dataset: ETTh1
     hparams: {windows: 96, pred_len: 96, horizon: 1}
     metrics: {mse: 0.40, mae: 0.30}
     source_type: paper
     citation: "DLinear (Zeng 2023)"
     url: "https://arxiv.org/abs/2205.13504"
     notes: ""
   ```
   Curated entries have no seed; they contribute one synthetic row with `seed: null`.

### Algorithm

```
for each result JSON:
    emit one Entry(model, task, dataset, seed, hparams, metrics, ...)

for each YAML entry:
    emit one Entry(..., seed=null, source_type="paper")

collect schema:
    tasks            = sorted unique Entry.task values
    datasets_by_task = {task: sorted unique datasets}
    models           = sorted unique Entry.model values
    hparams_by_task  = {task: sorted union of hparam keys seen for that task}
    # hparam_options_by_task_and_key = {task: {key: sorted unique values}}

write webapp/public/leaderboard_data.json
```

No aggregation happens here — every individual seed is an Entry. Aggregation is client-side.

## Frontend: React + Vite + TanStack Table + Tailwind CSS

### State Shape (`App.tsx`)

```ts
interface Filters {
  task: string | null;         // single-select
  datasets: string[];          // multi-select
  models: string[];            // multi-select
  hparams: Record<string, string | null>;  // {key: value | null}
}

interface ViewOptions {
  aggregate: boolean;          // true = group by (model,task,dataset,hparams), false = per-seed rows
  showStd: boolean;            // show ± std column suffix (only when aggregate=true)
  visibleMetrics: string[];    // which metric columns are visible
}
```

### `useFiltered.ts` — derives table rows from entries + state

1. Filter `entries` by active `Filters` (task, datasets, models, hparam dropdowns)
2. If `aggregate = true`:
   - Group by `(model, task, dataset, hparam_config_json_sorted)`
   - For each group: compute `{metric: {mean, std}}` across seed entries in the group
   - Emit one `AggregatedEntry` per group with `num_seeds` field
3. If `aggregate = false`:
   - Return filtered entries directly as rows
4. Sort by the active sort column; ties broken by model name

### `FilterSidebar.tsx`

- **Task** — radio buttons. Switching task:
  - Resets `filters.datasets`, `filters.hparams`
  - Shows only datasets from `schema.datasets_by_task[task]`
  - Shows only hparam controls from `schema.hparams_by_task[task]`
- **Dataset** — text search input + checkbox list (multi-select)
- **Model** — text search input + checkbox list (multi-select)
- **Hparams** — one `<select>` per hparam key for current task; option list from `schema.hparam_options[task][key]`; always includes `"All"` option (= no filter on that key)
- **Reset all** button — clears all filters

### `ToolBar.tsx`

- **Aggregate / Per-seed** toggle button
- **Show std** checkbox (enabled only when Aggregate is active)
- **Columns** dropdown — toggle visibility of individual metric columns
- **Export CSV** button — downloads filtered+sorted rows as CSV
- **Refresh** button (live server only) — calls `GET /api/refresh`, then re-fetches `leaderboard_data.json`

### `LeaderboardTable.tsx`

TanStack Table with:
- Column: `#` (1-indexed rank, not stored — derived from sort order)
- Column: `Model` — if `url` is set, renders as a link
- Column: `Task`
- Column: `Dataset`
- Column: `hparams` — rendered by `HparamBadges` as compact chips (e.g., `w96 p96 h1`). Clicking a chip sets that hparam filter in the sidebar.
- Columns: one per metric key present in filtered rows
  - Aggregate mode: `MetricCell` renders `0.38 ± 0.01` or `0.38` depending on `showStd`
  - Per-seed mode: plain float
  - Best value in column: green background; worst: faint red
- Column: `Seeds` (aggregate mode only) — shows `n=3`
- Column: `Source` — badge `local` or `paper`; hover shows citation tooltip

All columns are sortable. Default sort: first metric column. Sort direction determined by metric name:
- **Ascending** (lower is better): any key containing `mse`, `mae`, `loss`, `error`, `rmse`, `crps`, `wis`, `nll`
- **Descending** (higher is better): any key containing `accuracy`, `f1`, `auc`, `precision`, `recall`, `score`
- Unknown metrics: descending by default

Best-value highlight follows the same direction.

### `MetricCell.tsx`

Receives `{mean, std}` (aggregate) or `{value}` (per-seed), a boolean `isBest`, `isWorst`.
- Best: `bg-green-100 text-green-800 font-semibold`
- Worst: `bg-red-50 text-red-400`
- Std rendered as `<span class="text-gray-400 text-xs"> ± 0.01</span>`

## Live Server: `scripts/serve_leaderboard.py`

```python
from fastapi import FastAPI
from fastapi.staticfiles import StaticFiles
import subprocess, pathlib

app = FastAPI()
DIST = pathlib.Path("webapp/dist")

@app.get("/api/refresh")
def refresh():
    subprocess.run(["python", "scripts/build_leaderboard.py"], check=True)
    return {"status": "ok"}

app.mount("/", StaticFiles(directory=DIST, html=True))
```

Run: `uvicorn scripts.serve_leaderboard:app --reload`

The Refresh button in the UI calls `GET /api/refresh`, waits for 200, then `window.location.reload()`.

## Deployment

### Static (GitHub Pages)

```bash
python scripts/build_leaderboard.py        # writes webapp/public/leaderboard_data.json
cd webapp && npm run build                 # writes webapp/dist/
# push webapp/dist/ to gh-pages branch
```

`leaderboard_data.json` is committed into `webapp/public/` for the static build. Add a GitHub Actions workflow that runs both commands and pushes `dist/` to `gh-pages` on pushes to `main`.

### Local Live Server

```bash
python scripts/build_leaderboard.py        # initial data build
cd webapp && npm run build                 # build frontend
uvicorn scripts.serve_leaderboard:app      # serve on :8000
```

The Refresh button re-runs `build_leaderboard.py` and reloads the page.

## Testing

**`tests/leaderboard/test_build_script.py`**:
- `test_ingest_result_json` — feed one result JSON, verify Entry fields match
- `test_ingest_yaml_entry` — feed one YAML entry, verify `source_type="paper"`, `seed=null`
- `test_schema_hparams_by_task` — multiple Forecast runs with different windows/pred_len → `hparam_options` lists all observed values
- `test_no_aggregation` — output has one entry per input seed, not aggregated
- `test_idempotent` — running build twice on same inputs produces identical JSON

**Frontend** (Vitest + React Testing Library):
- `useFiltered.test.ts` — task filter removes entries from other tasks
- `useFiltered.test.ts` — aggregate mode groups seeds and computes correct mean/std
- `useFiltered.test.ts` — hparam filter `windows=96` removes entries with `windows=336`
- `MetricCell.test.tsx` — best value gets green class; std hidden when `showStd=false`
